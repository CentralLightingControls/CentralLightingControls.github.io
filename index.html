Central Lighting Controls designs, and develops software components for the control of entertainment lighting equipment. We are the behind the scenes people for the people who make stuff for the people behind the scenes.

Philosophy

Use standards when available.

If a standard is in place, we’ll use it. A ratified standard is always given preference to a defacto standard. The most glaring example is that we strongly prefer E1.31 “sACN” over Artnet. Not that Artnet isn’t great, but a standard is in place for transporting DMX style data over a network.

Components are lightweight.

The software components we create, run extremely efficiently. Communication with some components may at first seem limited, but these choices allow for minimal overhead and translation. For example the FadeEngine can operate over 10,000 concurrent fades on commodity hardware(1 GHz ARM Cortex-A8).

Mindful of history, not afraid of the new.

Many paradigms in lighting control have roots in very old systems and methods. We embrace this and are respectful of it. However, especially with technological solutions, the new might just be well suited to the problem. We use a programming language that is “new” by conventional standards, but when viewed through the lens of lighting control (historically and modern networked systems), a functional style with strong concurrency patterns makes a whole lot of sense. It also helps that it was designed for building the backend of some of the largest web based systems. (Go at Google)

Components

Our components are meant to be mixed and matched and integrated into other systems. We prefer to share data to and from our components using JSON or Go gob elements. A variety of other encoding types are easily implemented. The mechanics of the communication can also be easily configured to match your designs.

ShortHand

ShortHand is an HTML and Javascript based component and command syntax that is designed to facilitate web based console command entry. Key combinations are minimized, drastic hand shifting is reserved for the least frequent commands. Optimized for number pad entry, it also works great for tablet and smart-phone entries. ShortHand parses much of the command line before leaving the client browser, and uses web sockets to send JSON back to the host application. It is very tolerant of typos and accidental entries, so the user can continue to move quickly.

Monitor

Simply watches sACN traffic. Property values can be presented graphically through a built in web server, or retrieved as JSON via a restful API. Web pages views receive data through real time web-socket connections, updating graphs or presenting specific values. URLs are human readable and remember-able, ie: http://serveraddress/u1/bar presents a constantly updating bar graph of all slots in universe #1, http://serveraddress/u1/s1/line presents a “rolling” line graph and actual values for universe 1, slot 1.

The API is similarly straight forward. http://serveraddress/api/s1 will return a JSON array of all property values in universe 1 in the format {s: 1, v: 255}. While http://serveraddress/api/s1/vnot/0 will return an array of all values that aren’t at 0.

Recorder

Saves raw DMX style data, recording multiple universes simultaneously. The quantity of universes is dependent on the hardware available. E1.31 is fully supported, we can also configure to capture rs485 (standard DMX) depending on the hardware. Frames are timestamped relative to each other. Data can be saved raw as an array of frames with all bytes in the packet represented, or non-destructively optimized only presenting values that change. The Recorder is configurable to accept 16-bit groups of slots, each 16-bit group can be configured as big or little endian, ensuring compatibility with that odd piece of gear. Individual slots (or slot groups) can easily be returned from a stream. Recorder can also be configured to return the property values in several formats; 8 or 16-bit integers, 8 or 16 bit hex values, percentage (0–100), percentage floats (0.0 - 100.0), or normalized floats (0.0 - 1.0). All float values are 32-bits, the precision of returned values is easily configurable. 64-bit values can be made available for very specialized applications.

STMI

STMI (Stream To Move Instruction), can analyze a saved data set from the Recorder component and return move instructions. Move instructions may be decoupled from specific slot assignments and applied to just about any other type of slot grouping. For example an instruction set captured with 8-bit data can be easily reassigned and used on a 16-bit slot group. The format is simple and straight forward {v: 0.5, d: 10000} where v is the target value and d the duration, shown here as v: float(0.0 - 1.0), d: in milliseconds. fade to 50% over 10 seconds With STMI it is feasible to record a stream of DMX style values and extrapolate individual tracking style cues.

sCue

sCue or Simple Cue is a container for holding move fade tracking style cue data. Split fade times, linked cues, labeling, pause / resume execution, delay, and limited output are all available. The quantity of an sCue can easily be several thousands of instructions, depending on hardware. sCue is optimized for communication with our FadeEngine, but can be configured to deliver in many formats.

gCue

gCue or Group Cue, can queue many move instructions into one Cue. Many universes can be supported, with each individual property value changing on its own clock. Cue duration can be instant or spread out over days and weeks. gCues can be paused, resumed from any point relative to its start, faded in from current values, or enforce the gCue values at the beginning.

Both of our cue structures allow for filtering, reassignment of target device properties, and limiting. The can also be configured to allow “go to” commands, calculating all tracked values to that point.

FadeEngine

FadeEngine receives move instructions, delivering the computed values to its assigned collector. Multiple fade engines can be launched simultaneously. The FadeEngine is designed for last action, move fade, tracking styles of operation. Data can be delivered at regular intervals or only when values have changed. All fading values can be delivered in many different formats. Pause / Resume and dynamic transfer of output are supported.

DMXCollector

The DMXCollector receives changing values from a FadeEngine, or other system. Limited values, parked slots, translation to or from 16-bit slot groups and if needed slot level filtering / reassignment. Multiple DMXCollectors can be initiated with manual or timed cross fades possible between them. Allowing playback of 2(or more) streams each containing multiple universes of fading values, cross fading into each other. Collectors output is sent out at regular intervals to DMX hardware or our E1.31 components.

Other collectors supporting ACN style data are planned for the near future, including ACN DMP.

E1.31 (Streaming ACN)

Packs DMX style data structures into E1.31 packets and manages UDP connections. Full specification supported; duplicate data suppression and connection closing. Capable of generating standard compliant CID on the fly, if needed(not always recomended). Maximum data rate can be configured, defaults to ~44 frames a second per universe. 20 universes on low specification hardware is feasible.
